# Authoring Guide (UI / Logic)

このドキュメントは「ArsistでUIやロジックを作る方法」を“複数ルート”で整理し、どれがビルドに効くのか（Unity実機へ反映されるのか）を明確にします。

## 0. 重要: Source of Truth

- **プロジェクト（Project JSON）が唯一の真実**です。
- エディタの各タブ（UI/3D/Code）は「見せ方や編集方法」が違うだけで、最終的に **同じプロジェクトデータへ収束**します。
- Unityビルドは `manifest.json / scenes.json / ui_layouts.json` を入力として生成します。

## 1. UIを作る方法（複数）

### 1-1. ビジュアルUI（レイアウト編集）
- 対象: UIタブ（Canvas）
- 出力: `project.uiLayouts`（ツリー構造）
- 向いている: HUDや簡単な画面、配置調整
- 注意: 画像/動画などの外部アセットは **プロジェクト内Assetsへ取り込み** して参照します（相対パス）

### 1-2. コードUI（HTML/CSS/JS）
- 対象: Codeタブ
- 出力: `project.uiCode`（html/css/js）
- 同期:
  - `syncMode=two-way` の場合、UI⇄コードの相互変換が走ります。
  - どちらを最後に編集したかは `uiCode.lastSyncedFrom` に記録されます。

#### どこまでを保証するか（現実的な契約）
- **完全なHTML/CSS互換**は狙いません（Unity側生成に収束するため）。
- 代わりに、Arsistが理解できる最小サブセット（レイアウト、テキスト、ボタン、入力、画像、動画）を保証します。

### 1-3. UIテンプレ（プリセット）
- 目的: 「作り方が分からない」を最短で解消
- 例:
  - Floating Screen用の“設定パネル”
  - Head-locked HUD用の“ステータスバー”
  - World-anchoredの“ピン留めカード”

### 1-4. コンポーネント方式（将来拡張）
- React/Vueのように **UI部品をライブラリ化**して選べる方式
- ビルド時には「UIレイアウトツリー（宣言）」へ落とす

## 2. ロジックを作る方法（複数）

### 2-1. ビジュアルロジック（グラフ）
- 目的: 非エンジニアでもイベント駆動の挙動を組める
- 注意: “触れる”UIが未成熟なら、**動かせる最小セットが揃うまで導線は隠す**方針が安全

### 2-2. スクリプトロジック（JS/TS）
- 目的: 開発者が最短で書ける
- 実装パターン:
  1) **エディタ内プレビューでJSを実行**（Unityに行く前の検証）
  2) ビルド時に **C#へ変換**（限定DSL）
  3) Unity側にJSランタイムを入れる（重量級、非推奨）
- 推奨（現時点）: 2) の **限定DSL**（イベント/状態/HTTP/UI更新など）からC#生成

### 2-3. 外部入力（リモート入力）
- 目的: スマホ/PCから操作・イベント送信（デバッグにも必須）
- 最小構成:
  - UDPまたはWebSocketでイベント受信
  - 受け取ったイベントを「プロジェクトのイベントバス」へ流す

### 2-4. Webデータ（HTTP/WS）
- 目的: 「汎用性」を担保
- 必須要件:
  - CORS/認証/キャッシュ/オフライン考慮
  - Unity実機でのネットワーク権限/証明書

## 3. アセット（GLB/画像/動画）取り込みの原則

- **外部パス参照は禁止**（壊れる/共有できない/再現できない）
- すべて `ProjectRoot/Assets/...` へコピーし、プロジェクト内は相対パスで保持
- 表示側（Editor Preview / Unity生成）は `projectPath` を基準に解決

## 4. ビルドに何が効くか

- 3D: `project.scenes`
- UI: `project.uiLayouts`（＋必要なら `uiCode`）
- ロジック: `project.logicGraphs`（将来: 生成コードへ）
- AR設定: `project.arSettings`

## 5. すぐに「使える」ための推奨ワークフロー（MVP）

1. プロジェクト作成（テンプレ選択）
2. 3DにGLBを取り込み（Assetsへコピー）
3. UIにPanel/Text/Button/Imageを置く（画像はAssetsへ）
4. ビルド（Unityパス/出力先/デバイス選択）

---

次に実装していく場合は、上記の **2-2（限定DSL→C#）** と **2-3（リモート入力）** が「Unity/UEの上澄み」を取る上で最短で効きます。
