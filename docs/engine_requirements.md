# Engine Requirements: "Unity/UEの上澄み"を取るために必要なもの

目的: Arsistを「Unity/UEを置き換える」のではなく、**Unity/UEの“面倒”を吸収して、ARアプリを最短で作れる上澄みレイヤ**にする。

ここでは必要要件を大量に列挙し、実装優先度を付けます。

## 0. 大前提（勝ち筋）

- **ランタイムはUnity（当面）**: まず“実機に確実に出る”を勝つ。
- Arsistは **プロジェクト形式 + 生成/変換 + デバイス差分吸収 + デバッグ** に集中する。
- 重要: エディタで作ったものが「Unityに入った瞬間に別物になる」を絶対に避ける。

## 1. 必須（P0）: 実際に開発が成立する最低ライン

### 1-1. 安定したビルド・再現性
- 同じプロジェクトは同じ成果物が出る（入力→出力が決定的）
- Unityテンプレート完全性（ProjectSettings/Packages含む）
- ビルドのログ/失敗原因が分かる（エラーの要約・再実行）
- ビルド前に同期（UI/コード/シーン）を強制する

### 1-2. アセットパイプライン
- プロジェクト内Assetsへの取り込み（GLB/PNG/JPG/MP4等）
- リネーム/重複/参照切れの防止（ハッシュ命名やメタデータ）
- Unity側でのインポート設定（テクスチャ圧縮、動画設定）

### 1-3. 入力（デバッグ可能）
- **リモート入力**（PC/スマホ→グラスのアプリにイベント送信）
- 例: ボタン押下、値変更、座標指定、ログ取得
- ローカルネットで繋ぐだけで動く（UDP/WS）

### 1-4. Webデータ
- HTTP取得・JSON→UI表示の最小機能
- 認証（トークン）、HTTPS、CORS/証明書の扱い
- オフライン時の挙動

### 1-5. 視点/視野ガイド
- FOV・安全領域・距離（快適視聴距離）を可視化
- 3DoF/6DoF/Head-lockedで操作制約が変わるのを明示

## 2. 強い差別化（P1）: Unity/UEの上澄みっぽさが出る領域

### 2-1. 宣言的UI + データバインディング
- UI要素に「データソース」を紐付ける（state/http/remote入力）
- 例: `Text.content = state.user.name` / `Image.src = http("...")`
- これがあると“画面を作る=データと結ぶ”に近づく

### 2-2. 限定DSL（JS/TS風）→ C# 生成
- UnityのMonoBehaviourに落ちる最小仕様
- イベント駆動（onStart/onTick/onClick/onMessage）
- 状態管理（setState/getState）
- HTTP/WS/UDP（必要最小）
- ログ/計測

### 2-3. デバイス抽象化（Adaptersの格上げ）
- ビルドパッチだけでなく、
  - 入力差（ジェスチャ/ポインタ）
  - 表示差（解像度/色域/レンダリング設定）
  - 権限/OS差（Androidの権限、Intent）
 まで吸収

### 2-4. ライブリロード/ホットデプロイ
- Unityを再ビルドせずUIや設定だけ反映
- 実機に差分Push（ADB経由など）

## 3. 本格化（P2）: いわゆる“エンジン”に寄せる領域

### 3-1. エディタ拡張/プラグイン
- UI要素・ロジックノード・アセットインポータを追加できる
- スキーマ/型生成（TS型 + Unity側C#型）

### 3-2. 物理/アニメ/タイムライン
- ここはUnityに寄せて薄くラップするのが現実的

### 3-3. プロファイリング/レコーダ
- 実機FPS、CPU/GPU時間、GC、ネットワーク

## 4. “様々な方法で作成できる”を具体化するメニュー

UI:
- ビジュアル（レイアウト）
- コード（HTML/CSS）
- テンプレ（プリセット）
- コンポーネント（部品ライブラリ）
- Figma/画像→UI（将来）

ロジック:
- ビジュアルグラフ
- DSL（JS/TS風）
- 外部入力（WS/UDP）
- Web（HTTP/WS）
- Unityネイティブ（拡張/プラグイン）

## 5. 次にやるべき“現実的な順番”（提案）

1) Assets取り込みのUI導線（Image/Videoの参照ボタン）
2) リモート入力（WS or UDP）最小実装 + エディタから送信UI
3) データバインディング（Textにstate/httpを繋げる最小）
4) 限定DSL→C#生成（まずはonClick + setText + httpGet）
5) Adapterを入力/表示まで拡張

この順番で行くと「作り方が分からない」「汎用性がない」を最短で潰しつつ、Unity/UEの上澄み感が出ます。
